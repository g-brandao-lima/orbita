# v2.3 Growth Features - Milestone Proposal (v2)

**Status:** Aprovada pelo humano em 2026-04-21, pronta pra virar roadmap oficial
**Data:** 2026-04-21
**Depois de:** v2.2 UX Polish (shipped 2026-04-20, 299 testes passando)

## Mudanca desde v1 da proposta

- **Shared Groups removida:** publico-alvo pequeno, solucao com duas contas Google + grupos espelhados atende sem complexidade de many-to-many.
- **Cache Layer Travelpayouts adicionada como Phase 32:** desbloqueia SEO, prediction e reduz gargalo SerpAPI. Virou pre-requisito.
- **Multi-trecho (originalmente v2.4) trazido pra v2.3 como ultima fase:** "grupo dentro do grupo", roteiro encadeado (BR → Italia → Espanha → BR).

## Objetivo da milestone

Transformar Flight Monitor de ferramenta pessoal em produto com canal de aquisicao organica e base de dados propria, eliminando gargalo de API.

Meta de negocio: 10 usuarios ativos ao fim da v2.3 + arquitetura que suporte 100+ sem explodir custo.

## Sequencia de fases (ordem obrigatoria)

### Phase 31.9: Price Fidelity Hygiene (quick win)
**Goal:** Usuario entende que o preco e referencia, nao valor final de checkout.

**Success Criteria:**
1. Card e email trocam rotulo "preco" por "preco de referencia Google Flights".
2. Tooltip ou nota de rodape: "pode divergir ate 5% do valor final, bagagem e taxas nao incluidas".
3. fast-flights removido do codigo (scraping fragil, retorna vazio silenciosamente).
4. Todos os testes passam apos remocao.

**Esforco:** 30min. Separa responsabilidade de percepcao de preco antes da Phase 32 expor o cache pra mais gente.

### Phase 32: Cache Layer (Travelpayouts Data API)
**Goal:** 95% dos reads do dashboard vem de cache local, nao de API externa.

**Success Criteria:**
1. Cadastro Travelpayouts ativo (afiliado), token salvo em .env como TRAVELPAYOUTS_TOKEN.
2. Nova tabela `route_cache` (origin, destination, departure_date, return_date, min_price, currency, cached_at, source).
3. Cron APScheduler 6h/6h chama endpoint `prices_for_dates` pra top 500 rotas BR × 45 datas (~22k entradas, TTL 6h).
4. `flight_search.search_flights_ex()` consulta cache primeiro; SerpAPI so pra cache miss ou refresh de grupo ativo.
5. Quota SerpAPI mensal cai de >2000 pra <500 chamadas/mes com mesmo numero de usuarios.
6. Admin panel mostra metrica: cache hit rate % e quota Travelpayouts restante.
7. Regressao: todos os testes continuam passando, nenhum grupo perde historico.

**Risco:** Travelpayouts retorna preco cached 7 dias antigos ocasionalmente. Mitigacao: marcar `source='travelpayouts_cached'` e fazer refresh SerpAPI sob demanda no detalhe do grupo.

**Requirements:** CACHE-03, CACHE-04, CACHE-05

### Phase 33: Public Route Index (SEO)
**Goal:** Visitante do Google que busca "passagem GRU Lisboa" cai em pagina publica do Flight Monitor, nao em competidor.

**Success Criteria:**
1. Rota `/rotas/{ORIG}-{DEST}` renderiza publicamente (sem login) com preco mediano atual, historico 180d, melhores meses, CTA "monitore essa rota".
2. Dados servidos 100% do `route_cache` da Phase 32, zero chamada externa por pageview.
3. Sitemap.xml dinamico lista todas as rotas com >=30 snapshots acumulados.
4. Meta tags Open Graph + Twitter Card com imagem gerada dinamicamente mostrando preco atual.
5. robots.txt permite indexacao; rel="canonical" em cada rota.
6. 5 rotas seed com dados reais (GRU-LIS, GRU-MIA, GRU-MAD, GIG-LIS, VCP-LIS) publicadas antes do soft launch.
7. Link de redirect de compra e affiliate link Travelpayouts (monetizacao passiva).

**Depends on:** Phase 32 (cache e pre-requisito).

**Requirements:** SEO-01, SEO-02, SEO-03

### Phase 34: Price Prediction Engine
**Goal:** Dashboard mostra recomendacao acionavel "Compre ate DD/MM", nao so preco atual.

**Success Criteria:**
1. Regra deterministica combinando: janela otima (60-90 dias antes do voo) + delta vs media 90d + volatilidade da rota. Tres classes: "Compre agora", "Aguarde ate X", "Monitorar".
2. Cada recomendacao com justificativa de 1 frase ("preco 23% abaixo da media E voo daqui 75 dias").
3. Script `scripts/backtest_predictions.py` roda contra historico do `route_cache` (com dados da Phase 32, ja tem volume) e reporta hit rate por classe.
4. Email consolidado mostra recomendacao no topo.
5. Cartao ATP: meta interna de >60% hit rate retrospectivo antes de promover copy pra UI.
6. Sem ML. Regras primeiro, ML so com dado real >6 meses.

**Depends on:** Phase 32 (backtest precisa do historico agregado).

**Requirements:** PRED-01, PRED-02

### Phase 35: Onboarding Wizard (condicional)
**Goal:** Usuario novo sai da landing com grupo ativo em menos de 30s.

**Success Criteria:**
1. Apos OAuth, se usuario nao tem grupos, wizard 2 passos: destino + periodo aproximado.
2. Wizard cria grupo automaticamente e redireciona pro dashboard ja com dados (puxados do cache Phase 32).
3. Copy do dashboard e email revistos: sem jargao ("balde fechando" → "vagas acabando"), tom direto.

**Condicional:** so executar se Phase 33 trouxer >=3 usuarios novos organicos. Sem usuario real, nao vale a pena.

### Phase 36: Multi-Leg Trip Builder ("grupo dentro do grupo")
**Goal:** Usuario monitora roteiro encadeado: BR → Italia → Espanha → BR, cada trecho com sua janela de datas, sistema detecta melhor momento do ENCADEAMENTO completo.

**Success Criteria (a formalizar na fase de spec, momento de discuss-phase):**
1. Modelo de dados: grupo-pai + N trechos sequenciais (origin, destination, janela de datas, duracao min/max de stay).
2. UI pra criar/editar roteiro multi-trecho.
3. Busca respeita encadeamento temporal (data de chegada do trecho 1 >= data de saida do trecho 0 + min_stay).
4. Preco total do roteiro = soma dos trechos.
5. Sinal de compra aplicado sobre o total, nao trecho a trecho.
6. Detalhe: edge cases (trecho open-jaw, stopover gratis em hub, retorno diferente da ida) TBD na spec.

**Dependencies:** Phase 32 (cache com multiplos trechos) + Phase 34 (prediction precisa lidar com total, nao trecho).

**Nota:** regra de negocio sera definida via `/gsd:discuss-phase 36` quando chegar o momento. Hoje e placeholder.

**Requirements:** MULTI-01, MULTI-02
